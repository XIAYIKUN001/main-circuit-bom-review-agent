# Main Circuit BOM Review Agent

A no-dependency Node.js CLI for reviewing main-circuit BOMs before purchase or
PCB release.

The agent is intentionally generic. It does not contain project-specific
voltage, topology, transformer, or vendor assumptions. Put your own design
rules in JSON and keep the source code reusable for public GitHub projects.

## Features

- Reads BOM files exported as CSV or JSON.
- Maps different BOM column names through configurable field aliases.
- Tags BOM rows by risk area, such as high voltage, resistor, capacitor,
  semiconductor, transformer, connector, and protection.
- Applies configurable rules through a small JSON condition language.
- Writes Markdown or JSON reports.
- Uses only built-in Node.js APIs.

## Quick Start

```bash
cd bom-review-agent
npm test
node src/index.js examples/main-circuit-bom.csv --output report.md
```

Use a custom rule file:

```bash
node src/index.js examples/main-circuit-bom.csv \
  --rules examples/rules.main-circuit.json \
  --format markdown \
  --output report.md \
  --fail-on error
```

JSON output:

```bash
node src/index.js examples/main-circuit-bom.csv --format json --output report.json
```

## BOM Input

CSV input should contain one header row. Typical columns are:

```csv
Value,Description,Designator,Footprint,Quantity
IRFP460 500V MOSFET,Half bridge switch,Q1 Q2,TO-247,2
```

JSON input can be either an array of row objects:

```json
[
  {
    "Value": "IRFP460 500V MOSFET",
    "Description": "Half bridge switch",
    "Designator": "Q1 Q2",
    "Quantity": "2"
  }
]
```

or an object with an `items` array:

```json
{
  "items": []
}
```

For XLSX BOMs, export the sheet as CSV from Excel, KiCad, Altium, or your EDA
tool before running this CLI.

## Rule File

A rule file is JSON:

```json
{
  "name": "Main circuit review rules",
  "fieldAliases": {
    "part": ["Value", "Part Number", "MPN"],
    "description": ["Description", "Desc"],
    "designator": ["Designator", "RefDes"],
    "quantity": ["Quantity", "Qty"]
  },
  "rules": [
    {
      "id": "hv-rating-required",
      "severity": "error",
      "when": { "hasTag": "high_voltage" },
      "unless": {
        "textMatches": {
          "pattern": "\\b\\d+(?:\\.\\d+)?\\s*(?:kv|vdc|vac|v)\\b",
          "flags": "i"
        }
      },
      "message": "High-voltage related parts must show a voltage rating.",
      "recommendation": "Add the rated voltage, insulation rating, or a manufacturer part number."
    }
  ]
}
```

If no rule file is supplied, the CLI uses built-in generic main-circuit rules.

## Condition Language

Supported conditions:

- `fieldMissing`: canonical field name, such as `part`, `designator`, or
  `quantity`.
- `fieldPresent`: canonical field name.
- `fieldMatches`: `{ "field": "part", "pattern": "...", "flags": "i" }`.
- `textMatches`: regex matched against all row text.
- `hasTag`: risk tag generated by the agent.
- `quantityLessThan`: numeric threshold.
- `quantityNotInteger`: boolean.
- `all`: array of conditions.
- `any`: array of conditions.
- `not`: one condition.

Rules may contain `when`, `unless`, or `failWhen`:

- `when`: the rule is considered only when this condition is true.
- `unless`: report a finding when `when` is true and `unless` is false.
- `failWhen`: report a finding when both `when` and `failWhen` are true.

## Output Severity

Supported severities are `info`, `warning`, and `error`.

Use `--fail-on error` or `--fail-on warning` in CI. The process exits with
code 1 when a finding at or above that severity exists.
